Build an NIO HTTP Server, Prerequisite

Kumasuke's Blog
2018-07-12
Build an NIO HTTP Server, HTTP Protocol, Java, Java NIO, Non-blocking IO

Disclaimer: Although all articles of this series and all related codes are original, the basic idea of the nio server is taking this project on GitHub as a reference.

This is the first article of ‘Build an NIO HTTP Server’ series in which we will use Java NIO API to build a non-blocking HTTP server that serves static resource files from local. Inn addition, these are all done without any help of third party library (except for logging and testing).

In this prerequisite, let’s have a look at the related Java NIO API and the basic syntax of HTTP Requests and Responses.

SocketChannel and ServerSocketChannel

SocketChannel

SocketChannel is a NIO counterpart of java.net.Socket, which could also connect to remote machine via TCP/IP protocol. Besides, you could configure SocketChannel in non-blocking mode, which you are not able to do with Socket.

There are two ways to create a SocketChannel. You could either create a SocketChannel by open a connection to remote machine, or you could accept an incoming connection from a ServerSocketChannel. Let’s see an examples:

This method reads data bytes from the SocketChannel into the ByteBuffer we created formerly. From the point of view of the ByteBuffer, the ‘read’ action will cause ByteBuffer turning into ‘write’ mode. You should call flip to turn the ByteBuffer into ‘read’ mode.
The return value of read method returns the number of bytes read, or -1 if the channel has reached end-of-stream(which means the connection has been closed).

Same as the SocketChannel and Socket, ServerSocketChannel is a counterpart channel of java.net.ServerSocket, which listens on specified port for any incoming TCP connection.

Our Response to Request

As we have said, we only deal with first three methods (i.e. OPTIONS, GET, HEAD).
For OPTIONS Request, we’ll always return 200 OK with an additional Allow header which shows all methods that our server could deal with;
For GET Request, we will return 200 OK for successful get, 404 Not Found for not-found file and 500 Internal Server Error for file reading errors;
For HEAD Request, we will return almost same as what we return for GET Request with empty message-body and zero Content-Length;
For all the other not supported methods, we will always return 406 Method Not Allowed as status.
In addition, Server that represents the name and version of our server, Date that represents response generated time in RFC1123 format and Content-Length which shows the length of message-body.
