Is there any way to fix it?

Before Java 1.5, there was a few workarounds to this problem (such as using ThreadLocal), however, these methods are mostly tricky and inefficient.

As of Java 1.5, the memory model of Java has been modified and the semantics of volatile has been improved. With volatile employed, no reordering will be performed during the initialization and assignment to theInstance.

Eager Vs. Lazy

Now that you could write singleton pattern in Java correctly, whether in eager or lazy form, it’s the time to decide which form’s best.

The Simplest is the best, under any circumstance, you should consider eager initialization at first. While lazy initialization may save a lot of memory footprint and initialization time at startup, it increases the time of getInstance most of time at a cost.

If you really need the characteristics provided by lazy initialization, you should always taken the final version of double-checked locking idiom into consideration. Not only because of its correctness, but also it is a well-received version towards this form of singleton pattern.
